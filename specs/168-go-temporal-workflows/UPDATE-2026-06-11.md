# UPDATE 2026-06-11 — Round 3 / Trilha D (Temporal workflows)

Spec: **168-go-temporal-workflows**

## Mudança

`ApproveOutboundRequest` (aprovação direta do organizador, fora do fluxo de
votação) agora dispara o **OutboundExecutionWorkflow** após uma aprovação
bem-sucedida, alinhado ao `TemporalOutboundExecutionStarter` do Java.

### Detalhes

- Arquivo: `internal/usecase/outbound/approve.go`
- Setter nil-safe `WithTemporalStarter(portout.TemporalWorkflowStarter)`.
- Parâmetros do workflow (paridade Java):
  - **Task queue**: `OUTBOUND_EXECUTION_QUEUE`
    (`TemporalWorkerRegistry.OUTBOUND_EXECUTION_TASK_QUEUE`)
  - **Workflow ID**: `outbound-real-<outboundRequestId>`
    (`TemporalWorkflowIds.outboundPrimary`)
  - **Workflow name**: `OutboundExecutionWorkflow` (passado como string para
    evitar ciclo de import com o adapter Temporal — mesmo padrão dos use cases de
    payment).
  - **Args**: `{outboundRequestId, eventId, triggeredBy, reason}` —
    `triggeredBy` default `SYSTEM` quando o ator é vazio.
- **Não-fatal**: falha ao iniciar o workflow apenas loga `WARN`; a aprovação e a
  execução do provider seguem normalmente.

### Wiring

`cmd/server/main.go`:

```go
approveOutboundUC := ucoutbound.NewApproveOutboundRequest(
    outboundRequestRepo, eventoRepo, participantRepo,
    outboundTransferProvider, outboundAuditLogRepo,
).WithTemporalStarter(temporalStarter)
```

`temporalStarter` é o `ClientWorkflowStarter` real quando
`TEMPORAL_WORKER_ENABLED=true`, ou `NoopWorkflowStarter` caso contrário.

## Pendências

- O **worker** do `OutboundExecutionWorkflow` em Go ainda não está registrado
  (a Trilha B cobre a implementação do workflow). Até lá, com Temporal habilitado,
  o start usa o worker Java existente na mesma task queue (`OUTBOUND_EXECUTION_QUEUE`),
  garantindo interoperabilidade cross-backend.
